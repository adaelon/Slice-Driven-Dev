# 软件工程准则

> 这个 skill 不是一次性流程,是一组**在不同时刻被触发的子协议**。每条都有自己的触发条件、必须动作、禁令。
> **使用方法**:任务开始时扫一眼"触发词速查表",识别本次任务命中哪些协议,然后按命中协议办事。

---

## 启用条件(自动)

只要任务是下面之一,本 skill 默认激活:
- 写代码 / 改代码 / 重构 / 修 bug / 加功能 / 加测试
- 写技术文档 / 设计方案 / 决策落档 / 架构说明
- 与人讨论"该怎么做"且讨论会被沉淀

无需用户显式调用。但用户说出**触发短语**(见末尾速查表)时,**必须**走对应协议,不准跳过。

---

## 前置协议(Pre-flight):需求澄清

核心信念:**没有真实意图的对齐,后面所有切片都是猜**。

### §0 信心触发的澄清协议

**触发**:接到一个 SWE 任务(写代码 / 改代码 / 设计 / 写技术文档 / 落决策)之前,先自评——"我对用户**真实意图**的信心是多少?"

- 信心 **≥95%** → 直接动手,不打扰用户。
- 信心 **<95%** → 进入追问循环,直到 ≥95% 才出方案。

**协议**:
1. 挑**最影响方向的那一个**模糊点提问。判断标准:这一问的不同答案,会让方案走向截然不同的路;如果答案不会改变方案,就不值得问。
2. **一次只问一个问题**。不准一口气甩 3-4 个,因为后续追问应取决于上一个答案。
3. 用 **AskUserQuestion**(有明确候选时,带选项)或纯文本(开放问题时)。
4. 收到回答 → **重新自评信心** → 若仍 <95%,继续下一问;若 ≥95%,转入正式方案。
5. 给方案前,**简短复述一下我理解的意图**("两条都对齐了:X / Y"),让用户最后一次纠偏。

**禁令**:
- 不准达到 95% 之前就开始写代码或出长方案。
- 不准把"求确认"伪装成"澄清"——"我准备做 X,可以吗?"是请示,不是澄清;**澄清的问题必须能真正改变方向**。
- 不准问"我自己猜得到"的事(浪费用户时间);不准问纯偏好细节(颜色、命名)而忽略战略性歧义。
- 不准在用户已经把意图写得很清楚时硬找问题来问(过度澄清也是打扰)。

**自评信心常见的虚高陷阱**(出现以下任一情况,信心通常其实 <95%):
- 任务有**多种合理解读**,我心里默选了一种没核实;
- 用户没指定**关键参数**(范围、取舍优先级、边界);
- 任务和我经验里的常见模式**相像但不完全一样**;
- 涉及"这里要不要顺便做 X"的取舍;
- 用户给了一段含**两处以上不独立**歧义的需求。

**升降级**:
- 用户明确说"直接做 / 别问 / 你看着办" → 跳过 §0,带着最佳猜测进行,**在产出时显式标注"我假设了 X,如果不对回头改"**。
- 用户已经在请求里把意图写得很细 → 可能一上来就 ≥95%,直接动手。
- 追问超过 **4 轮**仍达不到 95% → 暂停追问,**把当前理解的多种解读列出来**让用户挑,而不是无限追问下去。

**示范**:这条 §0 协议自己是怎么定的——用户给的规则有两处歧义("95% 是触发还是停止"、"覆盖范围多大"),我先用 AskUserQuestion 问触发条件,根据答案再问覆盖范围,两轮后达到 ≥95% 信心,才开始落档。这就是协议的标准用法。

## A 组:小切片工作法

核心信念:**一次只动一刀,刀刀可验证**。多刀混在一起的失败模式是"出问题时不知道是哪刀的事"。

### A1. 切片声明协议

**触发**:动手写/改任何代码之前。

**协议**:动手前用 1-3 句话回答:
1. **本切片做什么**(一句话,动词开头)
2. **不做什么**(把容易混进来的相邻工作明确排除)
3. **完成的判据**(怎么算这刀干完了——通常是"X 测试通过"或"Y 命令产出 Z")

**禁令**:
- 不准在同一切片里同时做功能 + 重构;同时修两个不相关 bug;顺手"清理一下"无关代码。
- 不准用"顺便"。出现"顺便"两个字就是切片污染,把"顺便"的事拎成下一刀。

**为什么**:大切片回滚成本高、code review 无法定位、出 bug 时二分查不到根因。

### A2. 测试覆盖契约

**触发**:写了/改了任何可执行代码。

**协议**:
- **改了已有路径** → 跑相关测试,必须仍绿。
- **加了新路径**(新分支、新函数、新错误处理) → 写一条最小测试覆盖它。
- **修了 bug** → 先写一条能复现的失败测试,再修代码让它变绿(red-green)。
- **没有测试框架** → 切片完成前**告知用户**"这里没测试,我用 X 命令手动验证了 Y",不假装有覆盖。

**禁令**:
- 不准声称切片完成而测试没跑。
- 不准看到红测试就先"调整测试"——除非已经先确认是测试本身错了,且原因写清楚。
- 不准用 mock 替掉本应集成的依赖(参见 [feedback memory: 数据库集成测试不准 mock])。

### A3. 重构纪律

**触发**:用户说"重构"、或我自己识别到"想动结构但不动行为"的工作。

**重构的严格定义**:
> **功能不变 + 测试不变(全部仍绿) + 内部结构改变 + 改后更易理解和维护**。
> 任何一条不满足,就不是重构,得用别的协议。

**协议**:
1. 重构前**确认测试存在且全绿**;没测试就先补一道能锁住当前行为的测试(characterization test),再重构。
2. **小步走**:每一步都让测试保持绿。不允许出现"一连改十处,期间测试全红,最后一并修绿"。
3. 重构目标用**自然语言写一句**给自己看:"我想把 X 变成 Y,这样以后做 Z 会更容易"。看不懂这句话就先别动。
4. 重构产物的判据:**新读者(包括未来的我)能在不看历史的情况下理解这块代码**。

**禁令**:
- 不准把"加功能"伪装成"重构"。新增能力 = 功能,不是重构。
- 不准把"修 bug"伪装成"重构"。行为改了就不是重构。
- 不准重构+功能混在同一 commit/同一切片。

### A4. 长任务拆分(防上下文断裂)

**触发**:用户给的任务**预估会超过一个会话能跑完**——粗判据:涉及 3+ 文件大改 / 多步串行 / 要跑长流程 / 任务描述本身就是多段。

**协议**:
1. 在动手前,把任务拆成 **N 个独立可验的步骤**,每步:
   - 有明确的输入/产出
   - 有完成判据(能 run 起来 / 测试过 / 文档可读)
   - **不依赖会话内存,只依赖文件状态**(即下一会话从零接手能继续)
2. 用 TaskCreate 把这些步骤落成显式任务,**完成一步打一步勾**。
3. 每步结束**当场把成果落盘**(代码 commit-ready、文档段落写入、决策落档)。不允许"全做完再统一记录"。
4. 任何"我先全跑完再处理"的冲动 → 立刻拆细。

**禁令**:
- 不准把超长任务塞进一个连续的 tool-call 流。
- 不准依赖"上下文里那个我刚算出来的值"作为下一步的唯一来源——落盘或显式写在 task 里。

**为什么**:agent 上下文窗口有限,长任务跑到后半段时早期细节已被压缩,出错率飙升。**拆 + 落盘 = 让下一会话(或下一段思考)能从零接手**。

## B 组:与 AI 协作的中间语言

核心信念:**纯散文模糊,纯代码刚性,中间地带最高效**。

### B1. 精确中间语言

**触发**:与用户讨论"该怎么做"且讨论会变成行动(设计、接口、数据流、算法)。

**协议**:提案优先用**介于自然语言和编程语言之间的精确形式**,而不是纯散文。优先级:

1. **类型签名 / 函数签名**:`def schedule_next(history: list[Turn], budget: int) -> AgentId | "STOP"`
2. **伪代码**(关键控制流写出来,细节用自然语言):
   ```
   for each agent in roster:
       if moderator.should_speak(agent, context):
           yield agent.respond(context)
       if user.interrupted():
           break
   ```
3. **状态机 / 流程图**(文字版):`IDLE → PLAN → DEBATE ⇄ HUMAN_INTERJECT → SYNTHESIZE → IDLE`
4. **数据样例**(input → expected output 各举一条具体的):用真实样本而非"假设有个 X"
5. **spec 表格**(条件 × 结果):列穷举判断时

**禁令**:
- 不准用"我觉得可以做一个调度器,它会根据当前情况智能决定下一个发言者"这种**纯散文模糊提案**。
- 不准在还没有上面任一形式时声称"方案想清楚了"。
- 同时也不准上来就甩出完整代码——除非真到了写代码这一步,设计阶段应停在伪代码/签名。

### B2. AI 出输入,确定性工具出输出

**触发**:讨论"怎么验证一个东西对不对"、设计测试、设计反馈回路、设计 agent 奖励信号。

**协议**:
- **AI 适合做的**:出主意、生成测试输入、列边界 case、提建议、写代码、读代码做总结。
- **AI 不适合做的**:**判定对错**。判定要交给确定性工具——编译器、类型检查、测试 runner、linter、真实 API 返回值、规格比对。

**落实样式**:
- 测试设计:**AI 出输入用例**,**测试 runner 给绿/红**——不是"让 AI 看一下这个输出合不合理"。
- agent 系统的奖励信号:执行结果(代码跑通/测试过)做客观锚,**LLM-as-judge 只在没有客观锚时兜底**(参见 talk-agent 项目 §9.5)。
- 用户验收:**用真实数据跑一遍,看真实产出**,不是"让 AI 自评打个分"。
- 文档校对:**diff 工具 / linter / 链接检查器**,不是"让 AI 通读一遍说还行"。

**禁令**:
- 不准用 LLM-as-judge 作为唯一验证手段,除非任务**本质就没有客观判据**(纯主观偏好)。
- 不准说"我检查过了没问题"而没有跑确定性工具。
- 不准把"AI 输出 + AI 评分"的闭环当成可靠反馈——它会自循环。

## C 组:持续文档化

核心信念:**文档不是项目尾声写的总结,是工作过程的副产品**。每个切片都该掉下三类沉淀。

### C1. 决策落档(write-decision 协议)

**触发**:用户说"落档 / 写进文档 / 落进文档 / 把刚才的决策写下来"。**或**我自己识别到一段对话产生了一个会被未来引用的决策。

**模板**(硬性形状,不准超出):

```markdown
### §X.Y 标题(名词性,例:"调度器定级方式")

**决策**:一句话,不超过 30 字。

**否决**:
- A:一行理由。
- B:一行理由。
- (不超过 3 条)

**命门**:一行。              <!-- 可选——非显然的约束 -->
**何时回头**:一行。          <!-- 可选——重新审视的触发条件 -->
**展开**:[链接到原始讨论或更长论证]   <!-- 真要深挖的去处 -->
```

**禁令**:
- **不允许超过 2 级标题**(主标题 + 决策块内的加粗小标题,如此而已)。
- **不允许**写"落地为 / 这意味着 / 换言之 / 我们的做法是 / 命题如何拆解"这类铺陈词。
- **不允许**把对话过程当文档——文档的读者是几个月后的人,不是刚和我聊完的人。
- **不允许**整个决策段超过 15 行(不含展开链接)。
- 决策压不出 30 字 → 决策本身还不清,**先暂停落档,回去理清楚**。

**示范**:参见 talk-agent 项目 `docs\方案-AI微信群-需求与技术方案.md` §9.4 的精简版本(对比原版可看到长度 -60% 而要点一个不少)。

### C2. 代码链路

**触发**:每个切片(A1 定义的)完成时。

**协议**:在项目的 `docs\代码链路.md`(或等价文件)里,**追加一条**:

```markdown
## YYYY-MM-DD 切片名

**触达**:
- `path\to\file.py:Class.method` — 改了什么(一行)
- `path\to\other.py:func` — 加了什么(一行)

**入口**:从哪里能调到这条路径(用户操作 / API endpoint / 测试)。
**测试**:`tests\test_x.py::test_y` 覆盖了什么。
```

**为什么**:几个月后回来找"我当时为什么改了这个文件",从这条索引一查就到——不需要靠 git log 推。

**禁令**:
- 不准写"优化了若干处"这种模糊条目。每条必须精确到文件:符号。
- 不准等"项目结束统一整理"——切片完成时当场写。

### C3. 整体架构

**触发**:**结构变了才更新**。改增量代码不触发;新增模块、改模块边界、改主要数据流、引入新组件类型才触发。

**协议**:维护一份 `docs\架构.md`(项目级),含:
1. **组件图**(文字版或 mermaid):有哪些主要模块、谁调谁
2. **数据流**:主要操作的端到端流程(用 B1 的状态机/伪代码形式)
3. **关键决策的反向索引**:每个组件名后挂"为什么这样设计"→ 指向 C1 的决策记录

**禁令**:
- 不允许架构文档变成"代码注释合集"。它的作用是给**没看过代码的人**画蓝图。
- 不允许凭印象写——结构变更时**先读现状再更新文档**,确保和真实代码一致。
- 不允许架构文档过期超过一个切片周期。结构变了又懒得更新 → 切片不算完成。

### C4. SESSION_CHECKPOINT 热启动盘维护

**触发**:满足以下**任意一条**即触发:
- 一个大切片（A1 定义的）**完成并 commit** 后，准备结束当前会话或切换上下文；
- 用户说"刷新 checkpoint / 更新热启动盘 / 收口"；
- 预感上下文窗口即将撑满（距离截断还有 2-3 轮时）；
- 长任务中途搁置，下次会话需要从中间继续。

**协议**:整页**覆写**（不是追加）`SESSION_CHECKPOINT.md`，结构固定如下:

```markdown
# SESSION_CHECKPOINT — YYYY-MM-DD HH:MM

## 新鲜度自检
- 写入时最新 commit: <hash> <title>
- 读入时请对比 `git log -3`,若不一致以 git log 为准。

## 当前在做什么
一句话:正在做哪个大切片 / 功能,目标是什么。

## 下一步(可直接接手)
1. 第一步:具体动作(文件/函数/命令)
2. 第二步:…
(不超过 5 步;步骤必须是"能立刻执行"的原子动作)

## 未提交 / 未完成
- 文件或模块:状态说明(如"已改未测"、"测试红"、"待 commit")
- 无则写:无

## 冷启动读序
按顺序读这些文件能还原全局上下文:
1. `docs/技术方案-xxx.md` — 架构骨架
2. `docs/代码链路.md` — 改动账本
3. (其他必读文件)

## 本会话决策摘要
- §X.Y 决策名:一行结论(已落档到 docs/技术方案 §X.Y)
- (无决策则省略本节)
```

**禁令**:
- **不准追加**——每次都是整页覆写，旧内容全部替换，文件永远只反映最新状态。
- **不准写流水账**——不是"今天做了什么"，是"下一个陌生会话需要知道的最少信息"。
- **不准超过 80 行**——超出说明写了冗余，压缩到关键信息。
- "下一步"不准写"继续完成功能 X"这种模糊指令——必须是可立刻执行的原子动作（文件路径、函数名、命令）。
- 不准把"已落档到文档"的决策重复展开——摘要一行 + 指针足矣。

**为什么**:AI 会话无持久记忆，每次重开就是失忆。SESSION_CHECKPOINT 是人工外挂的热内存，写得好下次接手零摩擦；写得烂等于没写。

## 触发词速查表

| 用户说... 或 任务是... | 必须激活 | 不准跳过 |
|---|---|---|
| 任何 SWE 任务开始,意图有歧义 | §0 需求澄清 | 一次一个问题,信心 ≥95% 才动手 |
| "落档" / "写进文档" / "落进文档" | C1 决策落档 | 模板硬性形状 |
| "重构" | A3 重构纪律 | 先确认测试绿 |
| "做个切片" / "做这一刀" | A1 切片声明 | 声明输入/产出/判据 |
| "拆任务" / "分步走" / 长任务 | A4 任务拆分 | 落盘 + TaskCreate |
| "这个怎么验" / "怎么测" | A2 测试 + B2 工具验证 | 不靠 LLM 自评 |
| "怎么做" / "设计一下" | B1 中间语言 | 类型签名/伪代码 |
| 任何代码编辑 | A1 + A2 | 先声明切片,完成跑测试 |
| 新增模块/改边界 | C3 架构更新 | 当个周期内完成 |
| 切片完成 | C2 代码链路 | 当场追加索引 |
| 大切片完成 / 会话即将结束 / "刷新 checkpoint" / 上下文将断 | C4 SESSION_CHECKPOINT | 整页覆写,不准追加 |

## 和其他 skill 的关系

- 本 skill 不替代 `verify`、`code-review`、`run` 等动作型 skill;那些是**工具**,本 skill 是**做事的纪律**。两者协同:本 skill 规定"切片完成必须跑确定性验证",`verify` 可作为执行手段。
- 与 feedback memory 配合:memory 记"在这个项目里特别要注意 X",本 skill 给通用骨架。memory 是个性,本 skill 是共性。

## 自检清单(每次完成大任务前过一遍)

- [ ] 动手前对用户真实意图有 ≥95% 信心?是真有,还是虚高跳过了 §0?(§0)
- [ ] 这次工作分成了几个切片?每个都有声明的产出和判据?(A1 / A4)
- [ ] 改过的代码路径都跑了测试?新路径加了覆盖?(A2)
- [ ] 如果做了重构,是不是真重构(行为不变 + 测试不变)?(A3)
- [ ] 与用户的设计讨论用了精确中间语言,还是模糊散文?(B1)
- [ ] 验证手段是确定性工具,还是靠 AI 自评?(B2)
- [ ] 决策点都落档了?用的是 C1 模板还是又写成长篇大论?(C1)
- [ ] 代码链路追加了?架构如有变更也更新了?(C2 / C3)
- [ ] 大切片完成或会话即将结束时,SESSION_CHECKPOINT.md 整页覆写了?(C4)
  - "下一步"每条都是可立刻执行的原子动作(有文件/函数/命令)?
  - 未完成/未提交项列清楚了?

任何一条答"否"且没合理理由,任务不算完。
